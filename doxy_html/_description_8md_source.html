10 Arn Data Objects {#gen_arnobj}
11 ----------------
12 All objects are stored in a tree hierarchy and the naming is similar to typical
13 file systems, e.g. "//Measure/Water/Temperature/value".
14 
15 To get a handle to a folder, use a path ending with "/", e.g "//Measure/Water/".
16 
17 Folder names can be empty. In the above example, the first level folder is empty and the
18 second level folder is "Measure".
19 The empty folder name can also be referred as "@". Again, the example can equally be written
20 "/@/Measure/Water/Temperature/value". This "@" is typically used when an empty name is
21 unacceptable, e.g. in the tree viewer of ArnBrowser.
22 
23 Each part in a given path is dynamically added as needed, i.e. any path can be used without
24 explicitly creating each folder in advance.

27 ### Modes ### {#gen_arnobjModes}
28 _Mode_ change is a one direction process. Once a specific _mode_ is set, it can't be reset.
29 
30 If the ArnItem is in a closed state when the _mode_ change is done, the added modes will
31 be stored and the real _mode_ change is done when the ArnItem is opened to an
32 _Arn Data Object_. This implies that ArnItems can benefit from _mode_ settings before
33 being opened.
34 
35 If the _general mode_ change is done to a [shared](#gen_shareArnobj) object, the change
36 of _general mode_ is also done at the server and any connected clients.
37 
38 The following _general modes_ are available:
39 
40 * **BiDir** A two-way object, typically for validation or pipe.
41  See [bidirectional](#gen_bidirArnobj) objects.
42 * **Pipe** Implies _BiDir_ and all data is preserved as a stream during
43  [sharing](#gen_shareArnobj). Without _Pipe mode_, [sharing](#gen_shareArnobj) is
44  optimized to sync latest value and not all values in a stream.
45 * **Save** Sets the _Arn Data Object_ as persistent and any data assigned to it will be
46  saved. The persistent service must be started at the server.
47  See [persistent](#gen_persistArnobj) objects.
48 
49 Additionally there are some _sync modes_. These modes are used by the local client session
50 and are not shared with others.
51 The _sync modes_ must be set before the ArnItem is opened to an _Arn Data Object_.
52 
53 Following _sync_modes_ are available:
54 
55 * **Master** The _Arn Data Object_ (at client side) is set as _default generator_ of data.
56  Normally the server is the _default generator_ of data.
57  This makes difference when client connects or reconnects to the server.
58  The data from the _default generator_ is then used and synced.
59 * **AutoDestroy** The _Arn Data Object_ (at client side) is set up for auto destruction.
60  When the client closes tcp/ip, the server side will destroy the _Arn Data Object_ and
61  this will also be done at any connected clients.
62 
63 Note: It's convenient to always set all the needed modes before an ArnItem is opened or
64 an ArnItem is used as a template. See ArnItem::setTemplate().
65 <Br><Br>
66 
67 ### Naming conventions ### {#gen_naming}
68 These rules must not be obeyed, but are recommended, to get the most benefits of the
69 Arn echo system, like ArnBrowser.
70 
71 * First level folder not empty, e.g "/MyLocalFolder/Key/value", is a local path and is
72  not [shared](#gen_shareArnobj).
73 * First level folder empty, e.g. "//MyGlobalFolder/Date/value", is a global path and is
74  [shared](#gen_shareArnobj) to server and clients.
75 * When a leaf is used as an attribute, the following names are reserved:
76  + **value** the value of the above closest folder denotation, e.g. "Temperature".
77  + **set** allowed values and conversion to a more descriptive form, e.g. "0=Off 1=On".
78  + **property** like precision and unit, e.g. "prec=1 unit=°C".
79  + **info** like tool tips, e.g. "<tt>Standard UV radiation index</tt>".
80  + **help.**XXX like "help.html" contains help in xhtml format.

84 Bidirectional Arn Data Objects {#gen_bidirArnobj}
85 ------------------------------
86 A bidirectional _Arn Data Object_ is actually a double object, a twin.
87 Each part has its own path but their life span is depending on each other.
88 
89 One part is the normal "official" and the other part is _provider_.
90 The provider has an added "!" to the normal path, e.g. normal = "//Measure/Depth/value",
91 provider = "//Measure/Depth/value!".
92 
93 Data written to one part ends up in the other. When a provider slot is connected to the
94 provider part (ArnItem), the slot will receive "request" data from the normal part.
95 The provider slot processes the request data and writes the result to the same provider part.
96 This way the result will end up in the normal "official" part.
97 
98 This functionality can typically be used for data validation and limiting.
99 <Br><Br>
100 
101 ### Pipes ### {#gen_pipeArnobj}
102 _Pipes_ also use the [bidirectional](#gen_bidirArnobj) functionality. The two (twin) parts are then named _requester_ and _provider_.
103 
104 All data put into a pipe are part of a stream and as such will be fully transfered (syncronized) if they are [shared](#gen_shareArnobj) with a server and other clients.

108 Persistent Arn Data Objects {#gen_persistArnobj}
109 ---------------------------
110 The _server_ must use ArnPersist to support the persistance service. As a standard,
111 objects are stored in a SQLite database. It's also possible to store each object as a file.
112 
113 Any connected _client_ or the _server_ can make an _Arn Data Object_ persistent.
114 It's just to open an ArnItem to the object and change _mode_ to _Save_.
115 > ArnItem arnMaxLevel; <Br>
116 > arnMaxLevel.addMode( ArnItem::Mode::Save); <Br>
117 > arnmaxLevel.open("//config/Level/Max/value"); <Br>
118 
119 When the _Arn Data Object_ is set to _Save_ mode, it's automatically loaded by the
120 ArnPersist. At the _server_ this is instantly done. A _client_ has to wait for the value
121 to get synced from the _server_. It's convenient to use ArnDepend to get a signal
122 when the value is loaded and ready to use.
123 
124 When the _Arn Data Object_ is changed, it will be automatically saved by ArnPersist.
125 There is a delay from first change of the object until the saving is done,
126 see ArnItem::setDelay(). This allows for intensive updates of the object without choking
127 down the server with saving operations.
128 
129 It's possible to mark an object in the SQLite data base as _mandatory_. In this way the
130 _Arn Data Object_ is set as _persistent_ and gets loaded at start of ArnPersist.
131 <Br><Br>
132 
133 ### Saving objects in files ### {#gen_fileArnobj}
134 To use the _persistent_ storing of _Arn Data Objects_ in files, the _root_ directory is
135 set by: ArnPersist::setPersistDir(). This can also be combined with support of VCS
136 (version control system).See ArnPersist::setVcs(). Currently there is a support module
137 for _git_.
138 
139 In the _root_ directory and below, all (VCS) persistent files are stored.
140 The _root_ directory corresponds to the _root_ in Arn tree.
141 
142 > Example: _root_ directory is set to "/usr/local/arn_persist". There > is a file stored
143  at "/usr/local/arn_persist/@/doc/help.html". This > file will be mapped to Arn at
144  "//doc/help.html".
145 
146 Any files stored in the _root_ directory and below, get loaded into their _Arn Data Object_
147 with _mode_ set as _persistent_ at start of ArnPersist.
148 
149 The files get updated in a similar way to the data base update.

153 Sharing Arn Data Objects {#gen_shareArnobj}
154 ------------------------
155 A fundamental aspect of _Arn_ is that _Arn Data Objects_ can be shared. This is
156 centralized to the _Arn Server_, which stores all shared objects. It's still a
157 distributed model as each client and server has their own set of _Arn Data Objects_ that
158 operate independent of any connection.
159 
160 Each _Arn Client_ connects to the _Arn Server_ and decides which part of the
161 _Arn Data Object_ tree to be shared. <Br>
162 `ArnClient::setMountPoint("/share/")` will make the tree"/share/" shared. <Br>
163 This doesn't mean that everything in the shared tree at the server now will be available
164 at the client. The client has to create an _Arn Data Object_ in the shared tree.
165 The client can then decide the exact objects of interest. <Br>
166 `ArnItem::Open("/share/Test/value")` will open a shared object (in previous example).
167 
168 Note: Normally "//" is used for global (shared). See [naming conventions](#gen_naming).

172 RPC and SAPI {#gen_rpc}
173 ------------
174 ArnRpc is the basic functionality of RPC (Remote Procedure Call). ArnSapi implements SAPI
175 (Service Application Programming Interface) and is using ArnRpc as its base.
176 It's recommended to use ArnSapi which has a higher level model.
177 
178 The SAPI works by a model which can be described as RPC by _remote signal slots_.
179 The _provider_ is usually assumed to wait for a _requester_ to initiate the session
180 and then react to different remote calls from the _requester_.
181 However, this is full duplex, so any side can make a remote call at any time.
182 
183 A good example of the usage of SAPI is the _Arn Demo Chat_, which is included in the
184 source package of the _ArnLib_.
185 
186 ArnRpc uses [pipes](\ref gen_pipeArnobj) to communicate. The _pipes_ can be monitored
187 and receive test stimuli from the _Arn Browser_ program. The used
188 [protocol](#gen_rpcformat) is XString based and quite easy to handtype when common data
189 types are used. "$help" will give the syntax for the actual custom SAPI.
190 
191 A SAPI is setup by deriving the ArnSapi class to a new class that defines the
192 _custom SAPI_. This custom-declared class is included at both the _provider_ and
193 _requester_ ends.
194 The _custom SAPI_ class by itself doesn't implement any _services_. It's merely a hub for
195 connections to _external signals and slots_.
196 The base ArnSapi class automatically transfers all _custom signal_ (SAPI) calls to the
197 remote connected ends, which also have the ArnSapi derived class and that emits the
198 transfered signal.
199 
200 The provider connects the signals from custom SAPI that are prefixed with "pv_"
201 (as default) to each external slot that implements the services.
202 In the same way the _requester_ connects the signals prefixed with "rq_" to its external
203 "service" slots.
204 
205 When there is a naming pattern between the _SAPI services_ and the
206 _external signals and slots_, it's a great convenience to use ArnRpc::BatchConnect().
207 This saves a lot of QObject::connect() calls.
208 Also newly added services in the SAPI, that obey the naming scheme, will automaically be
209 connected to the newly matching _external signals and slots_ for implementation of the
210 _service_.
211 
212 An extended feature comparing to normal _signals_ is that the _SAPI
213 signals_ are _public_ and can be called by non-derived classes.
214 This makes it optional to use both _signal to signal_ connections or direct _signal_ calls
215 when issuing a RPC to the remote side.
216 
217 The _service_ slot can get the emitting _custom SAPI_ object by using normal
218 QObject::sender() functionality.

221 ### RPC and SAPI communication format ### {#gen_rpcformat}
222 The RPC calling has a basic format as XString (see XStringMap). The most generic form
223 is seen below. The type mark _T_ is "t" for writeable types and "tb" for binary
224 (non writeable) types.
225 > _funcname_ _T_=_type1_ a._label1_=_arg1_ _T_=_type2_ a._label2_=_arg2_ ... <Br>
226 > Example: put t=QString a.id=level t=int a.value=123 <Br>
227 > For calling: put( QString("level"), 123)
228 
229 Commonly used _types_ have a shorter form. The _types_ are:
230 > _int_, _uint_, _bool_, _ba (QByteArray), list (QStringList), and default is QString
231 
232 This can be used in previous example:
233 > put a.id=level int.value=123
234 
235 Or even shorter, skipping labels, when typed by hand:
236 > put level int=123
237 
238 List (QStringList) can be used. The _le_ is _list element_. All examples below will get
239 same resulting call.
240 > For a function: void test( QStringList lst, int num) <Br>
241 > test list=red green blu int=3 <Br>
242 > test list.lst=red green blu int.num=3 <Br>
243 > test list= le=red le=green le=blu int=3 <Br>
244 > test list=red le=green blu int=3 <Br>
245 
246 For special cases, like empty elements, the _le_ (list element) is needed. The example
247 below has a first empty element followed by "green".
248 > test list= le= green blue int=2
249 
250 The built-in call "$help" will give an automatically generated list of the present SAPI with the syntax for each available service.
